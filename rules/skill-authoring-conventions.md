# Skill Authoring Conventions

Rules for creating, updating, and maintaining SKILL.md files across the skills system.

## Patch Over Rewrite

**When updating an existing skill file, always prefer `Edit` (patch with minimal old/new strings) over full file rewrites.**

Why:

- Preserves surrounding context that wasn't part of the change
- Reduces token cost (sending a 3-line diff vs a 300-line rewrite)
- Makes review easier — the diff shows exactly what changed
- Prevents accidental deletion of sections not related to the update
- Matches the Hermes agent's documented best practice for skill self-modification

When to use full rewrite (Write tool):

- Creating a brand-new skill file
- Restructuring the entire skill (rare — discuss with user first)
- File is < 20 lines

## Auto-Generated Skills

Skills generated by `/meditate` Phase 6 or Claudia's `maybeGenerateSkill` are **drafts**:

1. Always include `— auto-generated from session on {date}` in the description
2. Set `model: sonnet` (not opus — auto-skills should be cheap to invoke)
3. Include a `## Notes` section with "Review and refine before relying on this skill"
4. Do not add to the skill-first routing table until manually verified

## Naming Conventions

- Skill directory: kebab-case (`deploy-oci-staging/`)
- Skill file: always `SKILL.md` inside the directory
- Name field in frontmatter: matches directory name
- Tags: lowercase, singular (`deployment` not `deployments`)

## Frontmatter Required Fields

Every SKILL.md must have at minimum:

```yaml
---
name: { kebab-case }
description: "{one-line with trigger phrases}"
user-invocable: true/false
context: fork/inline
model: haiku/sonnet/opus
effort: low/medium/high
allowed-tools: [list]
---
```

## Invocation Context Discrimination

Skills that can be called both by users and by other agents should include:

```yaml
invocation-contexts:
  user-direct:
    verbosity: high
    confirmDestructive: true
    outputFormat: markdown
  agent-spawned:
    verbosity: minimal
    confirmDestructive: false
    outputFormat: structured
```

## Tool Annotations

Declare safety hints for non-obvious tools:

```yaml
tool-annotations:
  Bash: { destructiveHint: true }
  mcp__memory__create_entities: { readOnlyHint: false, idempotentHint: false }
```

## Skill Lifecycle

1. **Draft** — auto-generated or manually created, untested
2. **Active** — manually verified, in the skill-first routing table
3. **Deprecated** — superseded by another skill, kept for reference
4. **Archived** — moved out of skills/ (or deleted if no longer useful)

Mark lifecycle in the description: `"(DRAFT) ..."` or `"(DEPRECATED) ..."`.
