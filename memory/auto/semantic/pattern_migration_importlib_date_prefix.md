Alembic migration files generated by `./apps/api/scripts/new_migration.sh` (UUID-based filenames) use a date prefix that starts with a digit — Python cannot import these via `import` or `importlib.import_module()` (invalid identifier). When writing unit tests for a specific migration's DDL contract, use this pattern:

```python
import importlib.util
import sys
from pathlib import Path
from unittest.mock import patch

_MIGRATION_FILE = (
    Path(__file__).parent.parent.parent
    / "alembic" / "versions"
    / "20260422_180000_2be29b416153_pgvector_0_8_upgrade.py"
)
_MIGRATION_ALIAS = "pgvec_migration_2be29b416153"

def _load_migration_module():
    if _MIGRATION_ALIAS in sys.modules:
        return sys.modules[_MIGRATION_ALIAS]
    spec = importlib.util.spec_from_file_location(_MIGRATION_ALIAS, _MIGRATION_FILE)
    mod = importlib.util.module_from_spec(spec)
    sys.modules[_MIGRATION_ALIAS] = mod
    spec.loader.exec_module(mod)
    return mod

migration = _load_migration_module()

def test_upgrade_ddl():
    with patch.object(migration, "op") as mock_op:
        migration.upgrade()
        # assert calls on mock_op...
```

**Key details:**
- Inject the alias into `sys.modules` before `exec_module` so intra-module imports (if any) resolve correctly
- Check `if _MIGRATION_ALIAS in sys.modules` to avoid re-loading in the same pytest session (prevents duplicate-index errors when `patch.object` is used)
- `patch.object(migration, "op")` works because `migration` is the live module object — `op` is imported at module-level in the migration file, making it a module attribute patchable by object reference

**When to use:** Whenever writing DDL contract tests for Contably's UUID-prefixed Alembic migration files.
